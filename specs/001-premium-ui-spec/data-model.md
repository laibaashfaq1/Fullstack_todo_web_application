# Frontend Data Model - Hackathon II Phase II Todo App

This document outlines the primary data structures the frontend application will consume and interact with, based on the backend API endpoints and the functional requirements.

## 1. Task Model (`Task`)

Represents a single todo item.

| Field       | Type      | Description                                     | Constraints       |
|-------------|-----------|-------------------------------------------------|-------------------|
| `id`        | `string`  | Unique identifier for the task (UUID).          | Read-only, generated by backend |
| `title`     | `string`  | Short description of the task.                  | Required, max 255 chars |
| `description`| `string`  | Detailed description of the task.               | Optional, multiline |
| `due_date`  | `string`  | Date when the task is due (ISO 8601 format).    | Optional          |
| `priority`  | `string`  | Importance level of the task.                   | `low`, `medium`, `high` |
| `status`    | `string`  | Current state of the task.                      | `pending`, `completed` |
| `user_id`   | `string`  | UUID of the user who owns this task.            | Read-only, set by backend |

## 2. User Model (`User` / Authentication Context)

Represents the authenticated user's basic information and authentication token. This model primarily reflects what the frontend needs to maintain user session and make authenticated requests.

| Field       | Type      | Description                                     | Constraints       |
|-------------|-----------|-------------------------------------------------|-------------------|
| `id`        | `string`  | Unique identifier for the user (UUID).          | Read-only         |
| `email`     | `string`  | User's email address.                           | Read-only         |
| `token`     | `string`  | JWT token for authenticated requests.           | Handled securely (httpOnly cookie) |
